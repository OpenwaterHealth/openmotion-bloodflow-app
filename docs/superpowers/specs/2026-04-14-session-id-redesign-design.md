# Session ID Redesign

**Date:** 2026-04-14
**Status:** Design — pending approval

## Background

The bloodflow app currently generates a new "session ID" at the start of every scan via `MOTIONInterface.newSession()` (called from `BloodFlow.qml`). This was an over-correction from Brad's original proposal: each scan becomes its own session, and the user cannot group multiple scans under one identifier.

Additionally, two session-related modals exist in the codebase but are unreferenced:

- `components/SessionModal.qml` — a combined Session ID + Notes dialog (from the Mar 25 refactor `e8663ba`, later reverted in `4317c65`).
- `components/UserSettingsModal.qml` — an older standalone Session ID editor, predating the Mar 25 refactor.

Both are dead files. Their mirrors in `src/openmotion_bloodflow/components/` are equally unreferenced.

## Terminology (new)

- **`userLabel`** — a short editable string (e.g. `owABC123`). Autogenerated at app startup as `ow` + 6 random uppercase alphanumerics. Editable by the user in the Scan Settings modal in research mode only. Persists across scans until the user changes it.
- **`sessionId`** — a composite, per-scan identifier of the form `{YYYYMMDD_HHMMSS}_{userLabel}`. Unique by construction because of the timestamp. One-to-one with a scan's output files (CSVs, notes).
- **Notes** — 1:1 with `sessionId`.

`sessionId` is an internal, scan-level concept — it is **not** exposed to QML as a `pyqtProperty`. It's computed inside the scan-start path and passed into `ScanRequest`.

## Behavior

### Research mode
- At startup, `userLabel` is autogenerated (e.g. `owA3B9X2`).
- User can open Scan Settings and edit `userLabel` at any time. The value persists across scans until changed.
- Each scan computes its own `sessionId = {timestamp}_{userLabel}` at scan-start.
- Notes are written to `{sessionId}_notes.txt`, 1:1 with the scan.

### Reduced mode
- `userLabel` is autogenerated at startup and used for all subsequent scans.
- The Scan Settings button is already hidden in reduced mode (`ButtonPanel.qml:131`), so the user has no UI path to edit `userLabel`. No additional gating needed.
- Each scan still computes a unique `sessionId` via timestamp.
- Notes behavior is identical to research mode.

### Notes lifecycle
- `_scan_notes` is an in-memory buffer on the connector; `_scan_notes_path` points to the currently-attached notes file on disk.
- Before any scan runs: `_scan_notes_path` is empty. The user can open the Notes modal and type; notes are held in memory.
- At scan-start: `_scan_notes` and `_scan_notes_path` are **cleared** (this replaces the side-effect that `newSession()` used to provide).
- At scan-end: connector writes `{sessionId}_notes.txt`, sets `_scan_notes_path`, appends a duration footer, and auto-opens the Notes modal.
- Between scan-end and next scan-start: any edit in the Notes modal saves back to the just-completed scan's notes file (via the `scanNotes` setter).
- Edge case: notes written before any scan — or written in the in-between gap and intended for the next scan — are discarded when the next scan starts. This is a documented trade-off of the 1:1 rule.

### History view
No change to on-disk filename format (already `{YYYYMMDD_HHMMSS}_{userLabel}_{side}_mask{XX}.csv`). The history modal already displays timestamp and label as separate columns, which together convey the `sessionId`.

## Implementation

### `motion_connector.py`

Both copies (root and `src/openmotion_bloodflow/`) must be updated.

1. Rename internal state `self._subject_id` → `self._user_label`.
2. Rename `generate_session_id()` → `generate_user_label()`. Implementation unchanged.
3. Replace the `sessionId` / `subjectId` `pyqtProperty` pair with a single `userLabel` `pyqtProperty` (read/write, notify `userLabelChanged`). Setter keeps the existing normalization (`ow` + uppercase alphanumerics).
4. Delete the deprecated `subjectId` / `subjectIdChanged` / `setSubjectId` / `getSubjectId` / `generate_subject_id` surface.
5. Delete `newSession()` entirely.
6. In the scan-start code path (where `ScanRequest` is built): compute `session_id = f"{scan_timestamp}_{self._user_label}"` and pass it as `ScanRequest.subject_id` (the SDK field name stays — only our naming changes). At the same point, clear `self._scan_notes = ""` and `self._scan_notes_path = ""` and emit `scanNotesChanged`.
7. History parsers (`get_scan_list`, `get_scan_details`) continue to accept both old (`scan_{id}_{ts}`) and new (`{ts}_{id}`) filename formats. `get_scan_details` returns both `userLabel` (the short piece) and `sessionId` (the full composite) in its result dict; legacy `sessionId` key meaning is updated, and any `subjectId` key is dropped.

### `components/ScanSettingsModal.qml` (and mirror)

Add a "Session" section at the top of the dialog with a labeled text field bound to `MOTIONInterface.userLabel`. Match existing styling. The field uses the same normalization the connector setter applies (so trailing spaces / invalid chars are normalized on commit).

### `pages/BloodFlow.qml` (and mirror)

1. Remove the `MOTIONInterface.newSession()` call at scan-start (line 123).
2. Change `subjectId: MOTIONInterface.sessionId` on the `ScanRunner` to `subjectId: MOTIONInterface.userLabel` (line 254). The ScanRunner passes this as the short label component; the connector composes the full `sessionId` internally.
3. The `sessionId` property on the BloodFlow page (if referenced) and the header display continue to read `MOTIONInterface.userLabel`.

### `main.qml` / `components/WindowMenu.qml` (and mirrors)

Rename the `sessionId` property binding to read from `MOTIONInterface.userLabel`. The header label text ("Session: owXXXXXX") continues to display the short userLabel.

### Deletions

- `components/SessionModal.qml`
- `components/UserSettingsModal.qml`
- `src/openmotion_bloodflow/components/SessionModal.qml`
- `src/openmotion_bloodflow/components/UserSettingsModal.qml`

Before deleting, grep confirms zero references in any `.qml`, `.py`, `qmldir`, or spec file (already verified in design phase).

## Out of scope

- Renaming `ScanRequest.subject_id` in the SDK (`openmotion-sdk`). That rename is a larger cross-repo change and not required for this redesign — we just pass the `userLabel` into the existing field.
- Adding a user-facing "new session" action. `newSession()` is removed; if a research user wants a new label, they edit the field in Scan Settings. No button is added.
- Changing on-disk filename format.

## Acceptance

- [ ] Startup generates a `userLabel`; header shows it.
- [ ] Research mode: Scan Settings modal has a User Label field; editing persists across scans; header updates live.
- [ ] Reduced mode: Scan Settings button hidden (unchanged); label is auto-generated and used for all scans.
- [ ] Each scan produces files named `{YYYYMMDD_HHMMSS}_{userLabel}_*` — identical to current behavior.
- [ ] Notes are cleared at each scan-start; saved at scan-end to that scan's `_notes.txt`; further edits in the post-scan window save to the same file.
- [ ] History view shows both the timestamp and the label.
- [ ] `SessionModal.qml` and `UserSettingsModal.qml` are removed from both copies.
- [ ] `newSession()` and `subjectId`-suffixed APIs are removed from the connector.
